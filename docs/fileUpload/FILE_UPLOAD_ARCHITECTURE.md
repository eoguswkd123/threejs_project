## 1. 전체 시스템 개요

### 시스템 구성도

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                              SYSTEM ARCHITECTURE                                │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│  ┌──────────────┐     ┌──────────────┐     ┌──────────────┐     ┌────────────┐ │
│  │   Frontend   │     │   Backend    │     │    Queue     │     │   Worker   │ │
│  │   (React)    │────▶│(Spring Boot) │────▶│ (RabbitMQ)   │────▶│  (Python)  │ │
│  │              │     │    [Java]    │     │              │     │            │ │
│  └──────────────┘     └───────┬──────┘     └──────────────┘     │            │ │
│         │                     │                                  │            │ │
│         │                     │ (상태 저장)                      │            │ │
│         │                     ▼                                  │            │ │
│         │             ┌──────────────┐                           │            │ │
│         │             │  PostgreSQL  │◀──────────────────────────│            │ │
│         │             │   (상태)     │                           │            │ │
│         │             └──────────────┘                           │            │ │
│         │                     │                                  │            │ │
│         │             ┌───────┴──────┐                           │            │ │
│         └────────────▶│   MinIO/S3   │◀──────────────────────────│            │ │
│          (glTF 로드)  │   (파일)     │ (원본/결과 저장)          └────────────┘ │
│                       └──────────────┘                                          │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘
```

### 동기/비동기 경계

| 구간 | 처리 방식 | 설명 |
| ---- | --------- | ---- |
| Frontend → Backend | **동기** | 업로드 요청, job_id 반환 대기 |
| Backend → Queue | **동기** | 메시지 발행 후 응답 |
| Queue → Worker | **비동기** | Frontend 응답과 무관하게 처리 |
| Frontend → Backend | **폴링** | 2초 간격, 최대 60회 (5분), COMPLETED/FAILED 시 중단 |

### 파일 크기 제한

| 계층 | 제한 | 설명 |
| ---- | ---- | ---- |
| Frontend | **500MB** | 빠른 검증, UX 피드백 |
| Backend | **500MB** | 보안 게이트 (우회 방지) |
| Worker | **500MB** | 처리 안정성 |

#### 용량 선정 근거

| 용량 | DXF 커버 | PDF 커버 | 업로드 시간 | 권장 |
|------|----------|----------|-------------|------|
| 50MB | 95% | 60% | 30초 | MVP |
| 200MB | 99% | 85% | 2분 | 초기 |
| **500MB** | 99.5% | **95%** | 5분 | **프로덕션** |
| 1GB | 99.9% | 98% | 10분 | Enterprise |

#### 대용량 파일 처리 요구사항

- **청크 업로드**: 10MB 단위 분할
- **진행률 표시**: 실시간 퍼센트 + 예상 시간
- **재시도 로직**: 청크 실패 시 3회 재시도
- **타임아웃**: 업로드 5분, Worker 처리 30분

### 단계별 설명

| 구간 | 단계 | 담당 | 설명 |
|------|------|------|------|
| **동기** | 1 | Frontend | PDF/DXF 파일을 POST 요청으로 전송 |
| | 2 | Java Backend | 파일 검증 후 MinIO/S3에 임시 저장 |
| | 3 | Java Backend | PostgreSQL에 job_id와 상태(PENDING) 저장 |
| | 4 | Java Backend | RabbitMQ에 작업 메시지 발행 (Enqueue) |
| | 5 | Java Backend | job_id 응답 반환 |
| **비동기** | 6 | Python Worker | Queue에서 작업 메시지 수신 (Dequeue) |
| | 7 | Python Worker | S3에서 파일 다운로드 (Read file) |
| | 8 | Python Worker | DB 상태 업데이트 (PROCESSING) |
| | 9 | Python Worker | DXF/PDF → glTF 변환 처리 |
| | 10 | Python Worker | 결과 glTF 파일 S3 저장 |
| | 11 | Python Worker | DB 상태 업데이트 (COMPLETED) |
| **폴링** | 12 | Frontend | 2초 간격으로 상태 폴링 (최대 60회) |
| *(5 이후 병렬)* | 13 | Java Backend | DB에서 상태 조회 후 응답 |
| | 14 | Frontend | COMPLETED 시 S3에서 glTF 로드 → 3D 렌더링 |

## 4. DXF 벡터 파이프라인

### DXF → 3D 변환 개요

DXF(Drawing Exchange Format) 파일의 벡터 데이터를 파싱하여 3D 모델로 변환합니다.

```
DXF → 엔티티 파싱 → 2D→3D 변환 → glTF 출력
```

### 파이프라인 다이어그램

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         DXF VECTOR PIPELINE                                  │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  ┌──────────────┐                                                            │
│  │   DXF 파일    │                                                            │
│  └──────┬───────┘                                                            │
│         │                                                                    │
│         ▼                                                                    │
│  ┌──────────────────────────────────────────────────────────────────────┐   │
│  │ STAGE 1: DXF 파싱 (ezdxf)                                            │   │
│  │                                                                       │   │
│  │ - 헤더 정보 추출 (단위, 스케일)                                        │   │
│  │ - 레이어 정보 추출                                                    │   │
│  │ - 엔티티 추출 (LINE, ARC, CIRCLE, POLYLINE 등)                        │   │
│  └──────────────────────────────┬───────────────────────────────────────┘   │
│                                 │                                            │
│                                 ▼                                            │
│  ┌──────────────────────────────────────────────────────────────────────┐   │
│  │ STAGE 2: 엔티티 변환                                                 │   │
│  │                                                                       │   │
│  │ - LINE → LineSegment (시작점, 끝점)                                   │   │
│  │ - ARC → Curve (세그먼트 분할)                                         │   │
│  │ - CIRCLE → Curve (360° 세그먼트)                                      │   │
│  │ - POLYLINE → Path (bulge 처리)                                        │   │
│  └──────────────────────────────┬───────────────────────────────────────┘   │
│                                 │                                            │
│                                 ▼                                            │
│  ┌──────────────────────────────────────────────────────────────────────┐   │
│  │ STAGE 3: 2D → 3D 압출                                                │   │
│  │                                                                       │   │
│  │ - Z=0 평면의 2D 좌표 → 높이(h) 압출                                   │   │
│  │ - 벽체: 두께(t) 적용하여 박스 생성                                     │   │
│  │ - 원형 요소: 실린더/파이프 생성                                        │   │
│  │ - 레이어별 높이/재질 매핑                                              │   │
│  └──────────────────────────────┬───────────────────────────────────────┘   │
│                                 │                                            │
│                                 ▼                                            │
│  ┌──────────────────────────────────────────────────────────────────────┐   │
│  │ STAGE 4: glTF 생성 (pygltflib)                                       │   │
│  │                                                                       │   │
│  │ - Mesh 생성 (엔티티별 3D 모델)                                        │   │
│  │ - Material 설정 (레이어별 색상)                                       │   │
│  │ - Scene 구성                                                          │   │
│  │ - glb 바이너리 출력                                                   │   │
│  └──────────────────────────────┬───────────────────────────────────────┘   │
│                                 │                                            │
│                                 ▼                                            │
│  ┌──────────────┐                                                            │
│  │   glTF/glb   │ ──────────────────────────────────▶ MinIO/S3 저장         │
│  └──────────────┘                                                            │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 단계별 설명

| Stage | 이름 | 입력 | 출력 | 예상 시간 |
| ----- | ---- | ---- | ---- | --------- |
| 1 | DXF 파싱 | DXF 바이너리 | 엔티티 목록 | ~0.5초 |
| 2 | 엔티티 변환 | 엔티티 목록 | 2D 좌표 배열 | ~0.3초 |
| 3 | 2D→3D 압출 | 2D 좌표 배열 | 3D Geometry | ~0.5초 |
| 4 | glTF 생성 | 3D Geometry | glb 파일 | ~0.7초 |
| **Total** | | | | **~2초** |

### 지원 엔티티

| DXF 엔티티 | 변환 결과 | 설명 |
| ---------- | --------- | ---- |
| **LINE** | 박스 (벽체) | 시작점→끝점을 두께와 높이로 압출 |
| **ARC** | 곡선 벽체 | 호를 세그먼트로 분할하여 압출 |
| **CIRCLE** | 실린더 | 원을 높이로 압출 (기둥 등) |
| **LWPOLYLINE** | 연속 벽체 | 연결된 선분/호 처리 (bulge 포함) |

### Bulge 처리

```
bulge = tan(θ/4)

θ: 호의 중심각
bulge > 0: 반시계 방향 (좌측으로 볼록)
bulge < 0: 시계 방향 (우측으로 볼록)
bulge = 0: 직선
bulge = 1: 반원 (90°)
```

## 5. PDF ML 파이프라인

### PDF → 3D 변환 개요

PDF 형식의 건축 도면을 ML 기반 분석을 통해 3D 모델로 변환합니다.

```
PDF → 이미지 추출 → 전처리 → ML 추론 → Geometry 생성 → glTF 출력
```

### DXF vs PDF 비교

| 항목 | DXF | PDF |
| ---- | --- | --- |
| **데이터 형식** | 벡터 좌표 | 래스터 이미지 |
| **처리 방식** | 직접 파싱 | ML 객체 탐지 |
| **정확도** | 높음 (원본 좌표) | 중간 (추론 기반) |
| **처리 속도** | 빠름 (~2초) | 느림 (~18초) |
| **ML 필요** | 불필요 | 필요 |

## 6. 저장 구조

### 데이터 분리

| 저장소 | 저장 내용 | 이유 |
| ------ | --------- | ---- |
| **MinIO/S3** | glTF/glb 파일 (3D 모델) | Three.js 네이티브 지원, CDN 캐싱 |
| **PostgreSQL** | 메타데이터 (job_id, status, result_url) | 상태 관리, 빠른 조회 |

### 왜 Geometry를 DB에 저장하지 않는가?

1. **크기**: Vertices 배열은 수천~수만 개의 좌표 (수 MB)
2. **호환성**: Three.js가 glTF/glb를 네이티브 로드
3. **성능**: CDN 캐싱으로 빠른 다운로드
4. **비용**: 오브젝트 스토리지가 DB보다 저렴

### S3 저장 경로

```
s3://cad-viewer-bucket/
├── temp/
│   └── {job_id}/
│       └── original.[dxf|pdf]    # 원본 (24h TTL)
│
└── results/
    └── {job_id}/
        ├── model.glb             # 3D 모델 (바이너리)
        ├── model.gltf            # 3D 모델 (JSON, 옵션)
        ├── metadata.json         # 변환 메타데이터
        └── preview.png           # 썸네일 (옵션)
```

### DB 스키마

```sql
upload_sessions
├── job_id (PK)           -- UUID
├── filename              -- 원본 파일명
├── file_type             -- 'DXF' | 'PDF'
├── status                -- PENDING | PROCESSING | COMPLETED | FAILED
├── temp_path             -- 임시 저장 경로
├── result_path           -- 결과 glTF URL
├── processing_route      -- 'VECTOR' | 'ML'
├── error_message         -- 오류 메시지 (실패 시)
├── created_at            -- 생성 시각
└── updated_at            -- 수정 시각
```
